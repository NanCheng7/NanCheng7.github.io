# SpringBoot 起步依赖原理分析

<dependencyManagement>：依赖版本锁定，会锁定依赖的坐标的版本，规定给子过程的坐标版本，使得子工程导入依赖时可以省略版本

小结

  • 在 spring-boot-starter-=parent 中定义了各种技术的版本信息，组合了一套最优搭配的技术版本
  • 在各种 starter 中，定义了完成该功能需要的坐标合集，其中大部分版本信息来自于父工程
  • 我们的工程继承 parent，引入 starter 后，通过依赖传递，就可以见到那方柏霓获得需要的 jar 包，并且不会存在版本冲突等问题

# SpringBoot 配置

如果 resources 里面同时拥有多个配置文件，那么优先级顺序：

application.properties > application.yml > application.yaml

如果高优先级文件和低优先级文件中配置了同样的属性，那么项目将采用高优先级文件中的属性配置，忽略低优先级文件中的属性配置。

小结：

  • SpringBoot 提供了 2 种配置文件类型：properties 和 yml/yaml
  • 默认配置文件名称：application
  • 在同一级目录下优先级为：properties > yml > yaml

# YAML

# 简介

TAML 全称是 YAML Ain’t Markup Language. YAML 是一种直观的能够被电脑是被的数据序列化格式，并且任意被人类阅读，容易和脚本语言交互，可以被支持 YAML 库的不同的编程语言程序导入，比如：C/C++,Rubby,Python,Java,Perl,C#,PHP 等。YML 文件是以数据为核心的，比传统的 xml 方式更加简洁。

YAML 文件的扩展名可以使用.yml 或者.yaml

###YAML 语法

  • 大小写敏感
  • 数据值前边必须有空格，作为分隔符
  • 使用缩进表示层级关系
  • 缩进是不允许使用 tab 键，只允许使用空格 (各个系统 tab 键对应的空格数目可能不同，导致层次混乱)
  • 缩进的空格数目不重要，只要同层级的元素左侧对其即可
  • # 表示注释，从这个字符一直到行尾，都会被解析器忽略

# YAML 数据格式

  • 对象 (Map)：键值对的集合

  • 数组：一组按次序排列的值

  • 纯量：单个的、不可再分的值

# Profile

我们在开发 SpringBoot 应用时，通常同一套程序会被安装到不同的环境，比如：开发、测试、生产等等。其中数据库地址、服务器端口等等配置都不同，如果每次打包时，都要修改配置文件，那么非常的麻烦。profile 功能就说来进行动态配置切换的。

  1. Profile 配置方式
    • 多 profile 文件方式
    • yml 多文档方式
  2. profile 激活方式
    • 配置文件
    • 虚拟机参数
    • 命令行参数

profile 小结

1.profile 是用来完成不同环境下，配置动态切换功能的

2.profile 配置方式

  • 多 profile 文件方式，提供多个配置文件，每个代表一种环境。

    • application-dev.properties/yml 开发环境
    • application-test.properties/yml 测试环境
    • application-pro.properties/yml 生产环境

  • yml 多文档方式：

    • 在 yml 种使用 — 分割不同配置

    • 在不同的文档页种使用

      进行文档分页的命名

3.profile 激活方式

  • 配置文件：在配置文件中配置： spring.profiles.active=dev
  • 虚拟机参数：在 VM options 指定： -Dspring.profiles.active=dev
  • 命令行参数：java -jar xxx.jar --spring.profiles.active=dev

# Spring 自动配置原理分析

# Condition

Condition 是在 Spring4.0 中增加的条件判断功能，通过这个功能可以实现选择性的创建 Bean 操作。

SpringBoot 提供了一个 @Conditional(value) 注解。value 存放的是实现了 Condition 接口的核心条件判断类，用来实现我们为创建 Bean 所需要拥有的条件判断。

Condition 是一个函数式接口，包含有一个 boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata) 方法，我们自定义的条件判断类就实现了这个接口的方法， matches 方法中 context 上下文对象。用于获取环境变量、IOC容器、ClassLoader对象 ， metadata 注解的源对象，可以用于获取注解定义的属性值 。我们根据需求编写一个动态的注解，能够根据我们是否导入的类坐标来创立 Bean , 则我们可以自定义一个注解 ConditionOnClass 。

如图可知我们自定义的注解中标注了 @Conditional(value) ，其中传值为我们自定义的核心判断类。且注解属性 value 为字符串数组，用于存放我们需要判断的坐标值。

如此就可以创建出自定义的 Bean 装载拦截器了

# SpringBoot 整合第三方技术

# 1、整合测试模块 Junit

由于在 Maven 项目中，Maven 的生命周期必须要用到测试，所以在 SpringBoot 的生成样例中一般会给我们自动导入测试的依赖。

注意：如果我们的测试类没有放在主启动程序的同级包或者子包下，会使得测试类无法获得容器对象，从而无法获得容器内的 Bean，导致测试出错。

解决方法：

  1. 更改测试类的路径从而使得其获得容器对象
  2. 在 @SpringBootTest(classes = SpringBoot04JunitApplication.class) 注解中主动导入主程序类的 class 文件
  3. 在测试类中用 @ContextConfiguration(classes =SpringBoot04JunitApplication.class) 标记，并在其中导入主程序类的 class 文件
