    List<String> configurations = SpringFactoriesLoader.loadFactoryNames(this.getSpringFactoriesLoaderFactoryClass(), this.getBeanClassLoader());
    Assert.notEmpty(configurations, "No auto configuration classes found in META-INF/spring.factories. If you are using a custom packaging, make sure that file is correct.");

      META-INF/spring.factories：自动配置的核心文件

结论：SpringBoot所有的自动配置都是在启动的时候扫描并加载：spring.factories所有的自动配置类都在这里面，但是不一定生效，要判断条件是否成立，只要导入了对应的Starter，就有对应的启动器了，有了启动器，我们自动装配就会生效，然后就配置成功了！

  1. springBoot在启动的时候，从类路径下``/META-INF/spring.factories`获取指定的值；
  2. 将这些自动配置的类导入容器，自动配置就会生效，帮我们进行自动配置
  3. 以前我们需要自动配置的东西，现在springBoot帮我们做了！
  4. 整个JavaEE，解决方案和自动配置的东西都在spring-boot-autoconfigure这个包下
  5. 它会把所有需要导入的组件，以类名的方式返回，这些组件就会被添加到容器
  6. 容器中也会存在非常多的xxxAutoConfiguration的文件(@Bean)，就说这些类给容器中导入了这个场景需要的所有组件;并自动配置 @Configuration，JavaConfig！
  7. 有了自动配置类，免去了我们手动编写配置文件的工作！

SpringBoot配置

如果resources里面同时拥有多个配置文件，那么优先级顺序：

application.properties > application.yml > application.yaml

如果高优先级文件和低优先级文件中配置了同样的属性，那么项目将采用高优先级文件中的属性配置，忽略低优先级文件中的属性配置。

小结：

  • SpringBoot提供了2种配置文件类型：properties和yml/yaml
  • 默认配置文件名称：application
  • 在同一级目录下优先级为：properties > yml > yaml

YAML

简介

TAML全称是YAML Ain’t Markup Language. YAML是一种直观的能够被电脑是被的数据序列化格式，并且任意被人类阅读，容易和脚本语言交互，可以被支持YAML库的不同的编程语言程序导入，比如：C/C++,Rubby,Python,Java,Perl,C#,PHP等。YML文件是以数据为核心的，比传统的xml方式更加简洁。

YAML文件的扩展名可以使用.yml或者.yaml

###YAML语法

  • 大小写敏感
  • 数据值前边必须有空格，作为分隔符
  • 使用缩进表示层级关系
  • 缩进是不允许使用tab键，只允许使用空格(各个系统tab键对应的空格数目可能不同，导致层次混乱)
  • 缩进的空格数目不重要，只要同层级的元素左侧对其即可
  • # 表示注释，从这个字符一直到行尾，都会被解析器忽略
server:
  port: 8080
  address: 127.0.0.1
  

name: abc

YAML数据格式

  • 对象(Map)：键值对的集合

    person:
  name: zhangshan
#行内写法
person: {name: zhangsan} #了解即可

  • 数组：一组按次序排列的值

    address:
  - beijing
  - shanghai
# 行内写法
address: [beijing,shanghai]

  • 纯量：单个的、不可再分的值

    msg1: 'hello \n world' # 单引号忽略转义字符
msg2: "hello \n world" #双引号识别转义字符

YAML参数引用

name: lisi

person:
    name: ${name} #引用上边定义的name值

Profile

我们在开发SpringBoot应用时，通常同一套程序会被安装到不同的环境，比如：开发、测试、生产等等。其中数据库地址、服务器端口等等配置都不同，如果每次打包时，都要修改配置文件，那么非常的麻烦。profile功能就说来进行动态配置切换的。

  1. Profile配置方式
    • 多profile文件方式
    • yml多文档方式
  2. profile激活方式
    • 配置文件
    • 虚拟机参数
    • 命令行参数

profile小结

1.profile是用来完成不同环境下，配置动态切换功能的

2.profile配置方式

  • 多profile文件方式，提供多个配置文件，每个代表一种环境。

    • application-dev.properties/yml 开发环境
    • application-test.properties/yml 测试环境
    • application-pro.properties/yml 生产环境

  • yml多文档方式：

    • 在yml种使用—分割不同配置

    • 在不同的文档页种使用

      spring:
  config:
    activate:
      on-profile: pro

      进行文档分页的命名

3.profile激活方式

  • 配置文件：在配置文件中配置：spring.profiles.active=dev
  • 虚拟机参数：在VM options指定：-Dspring.profiles.active=dev
  • 命令行参数：java -jar xxx.jar --spring.profiles.active=dev

Spring自动配置原理分析

Condition

Condition是在Spring4.0中增加的条件判断功能，通过这个功能可以实现选择性的创建Bean操作。

SpringBoot提供了一个@Conditional(value)注解。value存放的是实现了Condition接口的核心条件判断类，用来实现我们为创建Bean所需要拥有的条件判断。

Condition是一个函数式接口，包含有一个boolean matches(ConditionContext context, AnnotatedTypeMetadata metadata)方法，我们自定义的条件判断类就实现了这个接口的方法，matches方法中context 上下文对象。用于获取环境变量、IOC容器、ClassLoader对象,metadata 注解的源对象，可以用于获取注解定义的属性值。我们根据需求编写一个动态的注解，能够根据我们是否导入的类坐标来创立Bean,则我们可以自定义一个注解ConditionOnClass。

如图可知我们自定义的注解中标注了@Conditional(value)，其中传值为我们自定义的核心判断类。且注解属性value为字符串数组，用于存放我们需要判断的坐标值。

如此就可以创建出自定义的Bean装载拦截器了

@Enable*注解

在SpringBoot中提供了很多的enable开头的注解，这些注解都是用于动态启动某些功能的。而其底层原理是使用@Import注解导入一些配置类，实现Bean的动态加载。

当我们需要扫描外部工程的包时，首先需要在pom文件中导入坐标：

<!--获取其他项目坐标-->
<dependency>
   <groupId>com.NanCheng</groupId>
   <artifactId>SpringBoot-09-enableTest</artifactId>
   <version>0.0.1-SNAPSHOT</version>
</dependency>

但是根据@SpringBootApplication注解内源码我们发现：

它内部的ComponentScan注解扫描的范围仅包括引导类所在包及其子包，所以从其他项目引入的Bean无法被获取。那如果需要获取目标包，我们可以使用三种方式：

  1. 使用@ComponentScan扫描目标包，即

    package com.NanCheng.Sb;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.ConfigurableApplicationContext;

@SpringBootApplication
@ComponentScan("com.NanCheng.config")
public class SpringBoot09EnableApplication {
   
   public static void main(String[] args) {
      ConfigurableApplicationContext context = SpringApplication.run(SpringBoot09EnableApplication.class, args);
      //获取其他工程的bean
      Object user = context.getBean("user");
      System.out.println(user);
   }
}

  2. 使用Import注解，加载类，这些类都会被Spring创建，并放入IOC容器,即

    package com.NanCheng.Sb;

import com.NanCheng.config.UserConfig;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.ConfigurableApplicationContext;
import org.springframework.context.annotation.Import;
@SpringBootApplication
@Import(UserConfig.class)
public class SpringBoot09EnableApplication {
   
   public static void main(String[] args) {
      ConfigurableApplicationContext context = SpringApplication.run(SpringBoot09EnableApplication.class, args);
      //获取其他工程的bean
      Object user = context.getBean("user");
      System.out.println(user);
   }
}

  3. 可以对@Import注解进行封装，即我们无需关心目标Bean所在的位置，==只需要将目标包的项目通过pom文件导入，再导入目标Bean的@Enable注解即可获得相应的Bean,其底层原理依旧是@Import注解==

    package com.NanCheng.Sb;

import com.NanCheng.config.EnableUser;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.ConfigurableApplicationContext;
@SpringBootApplication
@EnableUser
public class SpringBoot09EnableApplication {
   
   public static void main(String[] args) {
      ConfigurableApplicationContext context = SpringApplication.run(SpringBoot09EnableApplication.class, args);
      //获取其他工程的bean
      Object user = context.getBean("user");
      System.out.println(user);
   }
   
}

    目标Bean提供的@Enable注解如下：

    package com.NanCheng.config;

import org.springframework.context.annotation.Import;

import java.lang.annotation.*;

@Target({ElementType.TYPE})
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Import(UserConfig.class)
public @interface EnableUser {
}

@Import注解

@Enable*底层依赖于@Import注解导入一些类，使用@Import导入的类会被Spring加载到IOC容器中。而@Import提供4种用法：

  1. 导入Bean
  2. 导入配置类
  3. 导入ImportSelector实现类，一般用于加载配置文件中的类
  4. 导入ImportBeanDefinitionRegistrar实现类

1、导入Bean

package com.NanCheng.Sb;

import com.NanCheng.pojo.User;
import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.context.ConfigurableApplicationContext;
import org.springframework.context.annotation.Import;

import java.util.Map;

@Import(User.class)
public class SpringBoot09EnableApplication {
   
   public static void main(String[] args) {

   }
   
}

我们直接在@Import()注解内直接加入获取的Bean的class。

MyImportBeanDefinitionRegistrar类源码：

package com.NanCheng.config;

import com.NanCheng.pojo.Role;
import com.NanCheng.pojo.User;
import org.springframework.beans.factory.support.AbstractBeanDefinition;
import org.springframework.beans.factory.support.BeanDefinitionBuilder;
import org.springframework.beans.factory.support.BeanDefinitionRegistry;
import org.springframework.beans.factory.support.BeanNameGenerator;
import org.springframework.context.annotation.ImportBeanDefinitionRegistrar;
import org.springframework.core.type.AnnotationMetadata;

/**
 * ImportBeanDefinitionRegistrar：导入注册Bean定义
 */
public class MyImportBeanDefinitionRegistrar implements ImportBeanDefinitionRegistrar {
   @Override
   public void registerBeanDefinitions(AnnotationMetadata importingClassMetadata, BeanDefinitionRegistry registry, BeanNameGenerator importBeanNameGenerator) {
      /*AbstractBeanDefinition：抽象的Bean定义*/
      AbstractBeanDefinition beanDefinition = BeanDefinitionBuilder.rootBeanDefinition(User.class).getBeanDefinition();
      registry.registerBeanDefinition("user",beanDefinition);
      
      AbstractBeanDefinition beanDefinition1 = BeanDefinitionBuilder.rootBeanDefinition(Role.class).getBeanDefinition();
      registry.registerBeanDefinition("role",beanDefinition1);
   }
}

我们想要获取对应的Bean，只需要在@Import()注解当中直接导入==注册了对应的Bean==且实现了ImportSelector的类即可。

@EnableAutoConfiguration注解

  • @EnableAutoConfiguration注解内部使用@Import(AutoConfigurationImportSelect.class)来加载类。
  • 配置文件位置：META-INF/spring.factories，该配置文件中定义了大量的配置类，当SpringBoot应用启动时，会自动加载这些配置类，初始化Bean。
  • 并不是所有的Bean都会被初始化，在配置类中使用的Condition来加载满足条件的Bean

1、整合测试模块Junit

由于在Maven项目中，Maven的生命周期必须要用到测试，所以在SpringBoot的生成样例中一般会给我们自动导入测试的依赖。

注意：如果我们的测试类没有放在主启动程序的同级包或者子包下，会使得测试类无法获得容器对象，从而无法获得容器内的Bean，导致测试出错。

解决方法：

  1. 更改测试类的路径从而使得其获得容器对象
  2. 在@SpringBootTest(classes = SpringBoot04JunitApplication.class)注解中主动导入主程序类的class文件
  3. 在测试类中用@ContextConfiguration(classes =SpringBoot04JunitApplication.class)标记，并在其中导入主程序类的class文件
